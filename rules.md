# PR WARDEN — RECONCILIATION RULES

Deterministic decision logic for the PR-vs-ticket drift gate. Every condition below is a boolean test computable from the pasted inputs alone — never a description requiring interpretation. Missing information is a deduction or a halt, never an inference. This discipline is inherited from CDLO, where live testing showed that every observed defect came from descriptive conditions being read differently across sessions; tests, not descriptions.

## REQUIRED INPUTS

1. **Ticket** — ID, title, description, acceptance criteria (if any), declared paths/components (if any), assignee (if any), status, links to related tickets (if any).
2. **PR payload** — title, body, author, commit list (id + message), changed-file list, dependency-manifest diff lines (if any). Generated by `scripts/extract_pr_payload.py` or pasted by hand.

TEST: ticket block absent OR contains no ticket ID → directive = `BLOCK`, reason "no baseline to measure against." The operator never reconstructs intent from the PR itself.

---

## RULE 1: BASELINE INTEGRITY SCORE (BIS) — start 1.00, subtract

Measures whether the ticket+PR pair is *verifiable*, independent of alignment.

| # | Boolean test | Deduction |
|---|---|---|
| B1 | Ticket description contains zero characters beyond the title AND zero acceptance criteria | −0.30 |
| B2 | PR body length < 50 characters | −0.20 |
| B3 | Per commit: message length < 10 characters (e.g. "wip", "fix") | −0.10 each, cap −0.30 total |
| B4 | Ticket status string is exactly `Closed` or `Done` at PR-open time | −0.20 |
| B5 | PR is marked Draft but the payload requests a merge verdict | −0.15 |

MUTUAL EXCLUSION: each deduction fires at most once per run except B3 (per-commit, capped). No single fact may be counted under two tests.

**Hard floor:** BIS < 0.70 → directive = `NEEDS-PROVENANCE` regardless of drift score. Rule 2 is still computed and reported.

## RULE 2: TICKET-DRIFT SCORE (TDS) — start 0.00, add, cap 1.00

Measures whether the PR does what its ticket says, and nothing else.

| # | Boolean test | Addition |
|---|---|---|
| D1 | The ticket ID string appears in none of: PR title, PR body, any commit message | +0.15 |
| D2 | Per acceptance criterion: the criterion is not addressed in the PR body (addressed = quoted, checkboxed, or referenced by its number) | +0.10 each, cap +0.30 |
| D3 | Per changed file: path is outside every path/component the ticket declares. SKIP TEST ENTIRELY if the ticket declares no paths/components — never infer scope. | +0.10 each, cap +0.30 |
| D4 | Per added dependency: a dependency-manifest diff line adds a package whose name appears nowhere in the ticket text | +0.10 each, cap +0.20 |
| D5 | Per commit: message references a DIFFERENT ticket ID (pattern: same tracker prefix, different number) | +0.15 each, cap +0.30 |
| D6 | Ticket has an assignee AND PR author login ≠ assignee login. SKIP if no assignee. | +0.10 |

**Thresholds:**

| TDS | Directive | GitHub mapping |
|---|---|---|
| ≤ 0.35 | `PASS` | success check |
| > 0.35 and ≤ 0.55 | `RE-ANCHOR` | neutral check + comment + label `drift:re-anchor` |
| > 0.55 | `BLOCK` | failure check |

## RULE 3: DIRECTIVE PRECEDENCE

1. `BLOCK` (missing ticket, TDS > 0.55, or any escalation)
2. `NEEDS-PROVENANCE` (BIS < 0.70)
3. `RE-ANCHOR` (TDS in 0.35–0.55 band)
4. `PASS`

Both scores are always reported, even when a higher directive short-circuits.

## RULE 4: ESCALATION TRIGGERS (audited FIRST, before all scoring)

### TRIGGER 1: DUPLICATE-AGENT SUBMISSION
TEST: two or more open PRs reference the same ticket ID AND share an identical title string AND their changed-file lists are identical.
ACT: directive string exactly `ESCALATION: COORDINATED DUPLICATE SUBMISSION — HUMAN REVIEW REQUIRED`. List the PR numbers and authors. Do not score any of them. (With agent fleets, identical parallel PRs signal uncoordinated agents racing one ticket — merging any of them without review compounds the collision.)

### TRIGGER 2: CONTRADICTORY BASELINE
TEST: the ticket's acceptance criteria contain two criteria where one requires X and another requires NOT-X (an explicit negation or mutually exclusive requirement stated in the ticket or a linked ticket marked duplicate/blocks).
ACT: directive string exactly `ESCALATION: BASELINE SELF-CONTRADICTION — TICKET OWNER REVIEW REQUIRED`. Quote both criteria verbatim in the log. Never pick a side; a contradictory baseline cannot adjudicate anything.

## SCORING NOTES (inherited from CDLO, stated up front)

The weights are policy defaults, not measured quantities — fixed, written, identically applied, tunable in `config.md`, never improvised per decision. BIS is a constructed integrity score, not a statistical confidence measure. This gate measures consistency between a PR and its ticket — documentation discipline, scope adherence, traceability. It does not measure whether the code works (CI's job), whether the implementation is good (review's job), or whether the ticket itself is wise (the team's job). It sits beside those gates, never instead of them.
